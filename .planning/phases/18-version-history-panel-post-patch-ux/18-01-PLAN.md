# Phase 18-01: Version History Panel & Post-Patch UX

## Objective
Implement a collapsible Version History bottom drawer showing timestamps, contextual descriptions, and status badges. Support preview, delete (with pointer chain repair), and restore features. Add a floating feedback toast after successful patches for instant reversion.

## Must Haves
- Successful patches must write new version rows to the database using `createPatchedVersion`.
- The Version History drawer must sit above the update input footer, scroll independently, and push sibling elements upward when open.
- Version descriptions must be contextually derived (user instruction for patches, 'Initial creation' for V1, 'AI Auto-Fix' for heals).
- Deletion of intermediate versions must automatically update their direct child's `parent_id` to repair the database version chain.
- The Post-Patch feedback toast must appear immediately after a patch lands, giving the option to revert instantly.

<task>

  <title>Implement version delete and pointer chain repair</title>
  <read_first>
    - src/services/artifactVersionDb.ts
  </read_first>
  <action>
    - Open `src/services/artifactVersionDb.ts`.
    - Implement `deleteVersion(dbId: string)`:
      - Fetch the version record to get `lineageId`, `parentDbId`, `versionNo`, and `isHead`.
      - Fetch any child version record (where `parent_id = dbId` and `lineage_id = lineageId`).
      - If a child exists, update its `parent_id` to point to the deleted version's `parentDbId`.
      - Perform the deletion query.
      - If the deleted version was the HEAD (`isHead === true`):
        - Find the preceding version in the lineage chain (maximum `version_no` remaining).
        - Update its `is_head` field to true.
  </action>
  <acceptance_criteria>
    - Deleting a middle version preserves parent linkages for remaining versions.
  </acceptance_criteria>
</task>
